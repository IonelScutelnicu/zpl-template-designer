# ZPL Template Designer

A visual, browser-based editor for creating Zebra Programming Language (ZPL) label templates. Design labels interactively on a canvas and generate production-ready ZPL code.

## Features

- **7 element types** ‚Äî Text, Text Block, Barcode (Code 128), QR Code, Box, Line, Circle
- **Dual preview** ‚Äî real-time Canvas rendering + Labelary API preview with ZPL syntax highlighting
- **Canvas interactions** ‚Äî click to select, drag to move, handle-based resize, arrow-key nudge
- **Element locking** ‚Äî lock elements to prevent accidental move, resize, or delete
- **Alignment tools** ‚Äî center horizontally/vertically, match to label dimensions
- **Undo/redo history** ‚Äî full history panel with named entries
- **Import/export** ‚Äî save and load templates as JSON files
- **Custom fonts** ‚Äî register custom ZPL fonts via `^CW` commands
- **Print settings** ‚Äî orientation, mirror, darkness, speed, home position, label top
- **ZPL warnings** ‚Äî Labelary API linter warnings with element-level attribution
- **Keyboard shortcuts** ‚Äî Tab cycling, Ctrl+Z/Y undo/redo, Delete, Ctrl+C/V copy/paste, arrow nudge

## Getting Started

```bash

npx serve . -l 3000
```

Open http://localhost:3000 ‚Äî no build step required (vanilla JS + Tailwind CDN).

## Element Properties

### Text
Position (X, Y), text content, font size (height), font width, font family, orientation.

### Text Block
Position (X, Y), text content (multi-line), font size, font width, block width, max lines, line spacing, justification (L/C/R/J), hanging indent.

### Barcode (Code 128)
Position (X, Y), barcode data, height, width multiplier, ratio, show interpretation line.

### QR Code
Position (X, Y), data, magnification factor, error correction level (H/Q/M/L), model.

### Box
Position (X, Y), width, height, thickness, color (B/W), corner rounding (0‚Äì8).

### Line
Position (X, Y), length, thickness, orientation (horizontal/vertical), color (B/W).

### Circle
Position (X, Y), diameter, thickness, color (B/W).

## ZPL Output Format

The application generates ZPL code in the following format:
```
^XA
^PW[width]
^PR[speed]
^PO[orientation]
~SD[darkness]
^LH[homeX],[homeY]
^LT[labelTop]
^CI28
^MTT
^CW[fontId],[fontFile]
^CF[fontId],[defaultHeight]
[element commands]
^XZ
```

Where:
- `^XA`/`^XZ` ‚Äî Start/End of label format
- `^PW` ‚Äî Print Width
- `^PR` ‚Äî Print Speed
- `^PO` ‚Äî Print Orientation
- `~SD` ‚Äî Media Darkness
- `^LH`/`^LT` ‚Äî Label Home / Top
- `^CW`/`^CF` ‚Äî Font Configuration
- Element commands are generated by each element's `render()` method

## Architecture

The application uses a modular architecture for maintainability and testability:

### Core Files
- `index.html` ‚Äî Main HTML structure and UI layout
- `main.js` ‚Äî Application entry point
- `app.js` ‚Äî Application orchestration and initialization
- `canvas-renderer.js` ‚Äî Canvas rendering orchestration
- `interaction-handler.js` ‚Äî Canvas interaction and drag/drop logic

### Element Definitions
- `elements/TextElement.js` ‚Äî Text element with font and orientation support
- `elements/BarcodeElement.js` ‚Äî Code 128 barcode element
- `elements/QRCodeElement.js` ‚Äî QR code element with error correction
- `elements/BoxElement.js` ‚Äî Rectangular box/border element
- `elements/LineElement.js` ‚Äî Horizontal/vertical line element
- `elements/CircleElement.js` ‚Äî Circle element with diameter and thickness
- `elements/FieldBlockElement.js` ‚Äî Multi-line text with wrapping and justification

### State Management
- `state/AppState.js` ‚Äî Centralized observable state store
  - Elements collection and selection
  - Label settings configuration
  - History management (undo/redo)
  - Event subscription system

### Services (Business Logic)
- `services/ElementService.js` ‚Äî Element CRUD operations
- `services/AlignmentService.js` ‚Äî Element alignment calculations
- `services/SerializationService.js` ‚Äî JSON serialization/deserialization
- `services/ZPLGenerator.js` ‚Äî ZPL code generation
- `services/TemplateManager.js` ‚Äî Template import/export file operations

### UI Components
- `ui/PropertiesPanelRenderer.js` ‚Äî Property form rendering for all element types
- `ui/ElementsListRenderer.js` ‚Äî Elements list sidebar rendering
- `ui/PropertyListenersManager.js` ‚Äî Property change event handling
- `ui/HistoryPanel.js` ‚Äî History panel UI and navigation
- `ui/CustomFontsManager.js` ‚Äî Custom font management UI

### Specialized Renderers
- `rendering/TextRenderer.js` ‚Äî Text element canvas rendering
- `rendering/FieldBlockRenderer.js` ‚Äî Field block with text wrapping
- `rendering/BarcodeRenderer.js` ‚Äî Code 128 barcode rendering
- `rendering/QRCodeRenderer.js` ‚Äî QR code pattern generation
- `rendering/BoxRenderer.js` ‚Äî Box/rectangle with rounded corners
- `rendering/LineRenderer.js` ‚Äî Line rendering (horizontal/vertical)
- `rendering/CircleRenderer.js` ‚Äî Circle rendering

### Utilities
- `config/constants.js` ‚Äî ZPL fonts, barcode patterns, configuration
- `utils/geometry.js` ‚Äî Geometry helper functions
- `utils/barcode-encoding.js` ‚Äî Code 128 encoding logic

### Design Principles

- **Single Responsibility** ‚Äî Each module has one clear purpose
- **Separation of Concerns** ‚Äî UI, business logic, and rendering are separated
- **Observable State** ‚Äî Central state management with event subscriptions
- **Dependency Injection** ‚Äî Services receive dependencies via constructor
- **Strategy Pattern** ‚Äî Specialized renderers for each element type

## Testing

The project uses Playwright for end-to-end testing with a dual-project setup optimized for performance and reliability.

### Test Organization

Tests are split into two projects to handle different execution requirements:

- **Core Tests** (~90 tests): Fast, parallel execution
  - Element creation/deletion, property updates, canvas rendering
  - ZPL generation, import/export, visual regression

- **API Integration Tests** (~25 tests): Sequential execution with rate limiting
  - API preview mode, canvas vs API parity testing
  - Respects Labelary API limits (3 requests/second)

### Running Tests

```bash
# Run all tests (both core and API)
npm test

# Run only core tests (fast, parallel)
npm run test:core

# Run only API tests (sequential with rate limiting)
npm run test:api

# Run tests in UI mode
npm run test:ui

# Debug tests
npm run test:debug
```

### Test Files

- Core tests: `*.spec.ts` (run in parallel)
- API tests: `*-api.spec.ts` (run sequentially)

API tests use automatic rate limiting (334ms between calls) to prevent hitting Labelary API limits.
